# 2026-06-29 框选交互修复

## 修改功能

- 框选触发范围从 Markdown 编辑器内部扩展到文档画布区域，正文右侧的大块空白处拖动也会出现蓝色框选区域。
- 文档画布作为框选定位容器，蓝色框和行级高亮会按页面画布坐标显示，不再局限于 `.ProseMirror` 正文节点。
- 保留原有图片缩放、图片复制和正文内部行级高亮逻辑。
- 鼠标松开后的真实选区改为按“被框到的文本行/图片块”计算，蓝框命中的某一行会选中整段对应块，减少整块矩形造成的误选。
- 图片块参与同一套块级命中规则，框到图片所在块时可以稳定选中图片块。
- 修正框选命中区域过宽的问题：每行只从自身内容起点向右扩展，不再把页面最左侧到该行之间的空白也当作命中区域。
- 松开鼠标后使用 DOM 位置、块内坐标和邻近节点共同反推 ProseMirror 顶层块边界，减少“没有选中”和“突然选中一大段”的情况。
- 选中内容后点击正文或画布空白处，如果没有形成拖动框选，会折叠当前选区并取消高亮显示。
- 正文编辑区内部的段落/标题右侧空白和 ProseMirror 根空白不再启动自定义蓝框框选，交还给 Milkdown/浏览器原生文字选择，避免拖动后误选到上方整片内容。
- 修复当前没有文字选区时，从画布空白处蓝框选择内容后偶发从文档开头/首页一路选到目标区域的问题；块位置解析现在优先使用视觉坐标，并拒绝“非首个块却解析为位置 0”的异常结果。
- 重构蓝框框选命中流程：拖动时的蓝色行高亮和松开后的真实选区现在共享同一份 `MarqueeHit` 结果，避免“看起来选中”和“实际选中”使用两套不同计算。
- 框选应用真实选区时优先通过 ProseMirror 顶层节点反查对应 DOM 块，只有校验通过后才使用 DOM 顺序范围，减少 `posAtCoords` / `posAtDOM` 在空白和块边缘处解析到后续块的问题。
- 选中文字后的格式菜单改为和横向工具栏一致的暖色浮层样式，避免菜单纯白背景和工具栏视觉割裂。
- 格式菜单打开后会在文档滚动或窗口尺寸变化时重新定位，不再停留在打开时的旧屏幕坐标。
- 文档目录高亮改为跟随编辑区滚动：根据视口顶部附近内容所属的最近标题常驻标记当前目录项。

## 说明

- 这次修复针对桌面端文档右侧留白区域；手机端仍以正文触控编辑为主，不额外启用大面积框选入口。

## 已知问题

- 蓝框框选逻辑仍存在未完全解决的问题：某些情况下蓝框视觉覆盖了多行内容，但松开鼠标后的真实选区可能没有完整匹配视觉范围，或偶发选区范围异常。该问题暂时标记保留，后续需要继续梳理 Milkdown/ProseMirror 选区映射逻辑后再处理。
